# ?? Skill Judge v3 — 技能化判定引擎（含进化审批）

## Contract

1. **只判定不创建** — 本技能只输出判定结果和建议，不自行调用 skill 管理 API
2. **可解释** — 每个评分维度必须附理由（一行），不可只给数字
3. **可复现** — 相同输入 → 相同输出
4. **输入**：AGENTS.md 门禁 5 的 3 问提取（决策 + 新发现 + 下轮记住）+ 任务性质描述
5. **输出**：六维评分 + 路由分类结论 + 建议动作 + 审批结论
6. **审批合一**：v3 吸收了原 skill-evolution-approval 的审批机制，改为单层决策（主人是唯一审批人）

---

## 技能层级（用于路由优先级和评分阈值）

| 层级 | 编号 | 说明 | 加载方式 | 清理策略 | 最低创建分 |
|:----:|:----:|------|:--------:|:--------:|:--------:|
| 系统核心 | L1 | Agent 框架自身依赖 | 始终加载 | 永不清理 | ≥60 |
| 工作流库 | L2 | 标准操作流程、怎么做 | 按阶段触发 | 手动审核 | ≥65 |
| 通用工具 | L3 | 文件转换、搜索、写作 | 按触发词匹配 | >180 天无触发告警 | ≥70 |
| 领域专用 | L4 | 管线专属能力 | 按上下文加载 | >90 天无触发清理 | ≥60 |
| 实验/临时 | L5 | 刚创建未经充分验证 | 最低优先级 | 90 天自动过期提案 | ≥50 |

## 六维评分矩阵

### 维度定义

| 维度 | 代号 | 权重 | 说明 | 测量方式 |
|:----:|:----:|:----:|------|:--------:|
| **问题严重性** | S | 25% | 不用这个技能会多惨？ | 评估单次节省的时间/风险 |
| **独立性** | I | 20% | 能否切割为单一职责？ | 评估与现有技能的耦合度 |
| **触发词区分度** | D | 15% | trigger 在已有池中是否可命中？ | wiki_search 实测命中率 |
| **创建成本** | C | 15% | 编写 SKILL.md + 依赖成本 | 评估工具依赖/外部数据需求 |
| **知识成熟度** | M | 10% | 该知识沉淀了几次？ | 门禁 5 出现次数 |
| **目标层级** | L | 15% | 适合放哪一层？ | 按上述层级定义映射 |

### 评分指南

#### S — 问题严重性（25%）

| 分数 | 标准 |
|:----:|------|
| 1-2 | 不用技能也能轻松搞定，只是多打几个字 |
| 3-4 | 不用技能需要搜索/翻文档，耗时 ≤5 分钟 |
| 5-6 | 不用技能需要手动排查/试错，耗时 5-30 分钟 |
| 7-8 | 不用技能可能遗漏/出错/重复踩坑，耗时 30 分钟+ |
| 9-10 | 不用技能可能造成破坏性后果，或需要数小时人工 |

#### I — 独立性（20%）

| 分数 | 标准 |
|:----:|------|
| 1-3 | 与其他技能高度耦合，不可单独执行 |
| 4-6 | 部分独立，需少量外部输入 |
| 7-8 | 完全独立，无外部依赖 |
| 9-10 | 自包含，可离线执行 |

#### D — 触发词区分度（15%）

**测量方式**（必须实际执行，不可估算）：

```powershell
# 对每个计划 trigger 词执行 wiki_search
# 统计结果中与计划技能语义重叠的技能数
# 取所有 trigger 词的平均重叠数
```

| 分数 | 标准 |
|:----:|------|
| 1-3 | 平均重叠 ≥4 个技能（语义拥挤） |
| 4-5 | 平均重叠 3 个技能 |
| 6-7 | 平均重叠 1-2 个技能 |
| 8-9 | 平均重叠 0 个技能，但有近似匹配 |
| 10 | 完全独有 trigger 词，无近似匹配 |

#### C — 创建成本（15%）

| 分数 | 标准 |
|:----:|------|
| 1-3 | 需外部 API/多工具依赖/复杂环境配置 |
| 4-6 | 需引用已有知识库，中等复杂度 |
| 7-8 | 纯文本指令，少量工具声明 |
| 9-10 | 纯文本描述，无工具依赖 |

#### M — 知识成熟度（10%）

| 分数 | 标准 |
|:----:|------|
| 1-2 | 门禁 5 首次出现，未经验证 |
| 3-5 | 出现 2-3 次，但操作模式未稳定 |
| 6-7 | 出现 3-5 次，操作模式基本稳定 |
| 8-9 | 出现 >5 次，操作模式稳定 |
| 10 | 已形成固定操作模式，多次验证无误 |

#### L — 目标层级（15%）

| 目标层 | 分数 | 条件 |
|:------:|:----:|------|
| L1 | 10 | 系统核心能力，agent 框架依赖 |
| L2 | 8-10 | 稳定流程，跨场景可共享 |
| L3 | 7-10 | 通用工具能力 |
| L4 | 6-10 | 领域专属，管线特定 |
| L5 | 5-10 | 实验性，不确定性高 |

### 加权公式

```
总分 = S×0.25 + I×0.20 + D×0.15 + C×0.15 + M×0.10 + L×0.15
```

### 层感知阈值

| 目标层 | 最低通过分 | 说明 |
|:------:|:--------:|------|
| L1 | ≥60 | 核心层门槛最低 |
| L2 | ≥65 | 工作流需一定成熟度 |
| L3 | ≥70 | 通用工具需高区分度 |
| L4 | ≥60 | 领域层门槛灵活 |
| L5 | ≥50 | 实验层最宽松 |

|:----:|:----:|------|
| ≥80 | ?? 高 | 可创建 |
| 50-79 | ?? 中 | 可追加或修改 |
| <50 | ?? 低 | 仅知识库，不技能化 |

## 路由节点（Step 2.5）

在评分完成之后、创建/追加动作之前插入。**路由优先于分数**。

### 路由决策树

```
评分完成 → 总分 N/100
  │
  ├─ 低于目标层阈值 → 仅知识库，退出
  │
  └─ 高于或等于阈值 → 执行路由检查
      │
      ├─ [A] 无匹配
      │   → 直接创建
      │
      └─ [B] 有匹配（命中 ≥1 个已有技能）
           → 执行关系分类
```

### 关系分类

| 关系 | 代号 | 判断条件 | 动作 |
|:----:|:----:|---------|:----:|
| **补充** | S | 新知识 ⊆ 已有技能（方向一致，范围更大） | `skill_manage patch` — 追加触发词/工具 |
| **校正** | C | 新知识与已有技能存在事实冲突，且新证据更强 | `skill_manage patch` — 替换/修正 |
| **修改** | M | 边界不同或核心逻辑不同需重定义范围 | `skill_manage patch` — 重写部分章节 |
| **完全覆盖** | O | 新知识 ≈ 已有技能，无新增价值 | 跳过，仅知识库，标注 `[路由:完全覆盖]` |
| **不确定** | U | 无法明确判断 | 输出候选 + 等主人确认 |

### 路由规则优先级

```
路由判定「覆盖」→ 跳过（分数再高也不创建）
路由判定「不确定」→ 输出候选 + 等主人
路由判定「补充/校正/修改」→ 执行 patch
路由判定「无匹配」→ 按原始分走 create/跳过
```

## 审批机制（v3 新增，原 skill-evolution-approval 审批流简化）

v3 将原多层审批流简化为**单层决策**：主人是唯一审批人。

| 分数区间 | 决策 |
|----------|------|
| ≥ 80 | 自动批准（直接创建/修改） |
| 50-79 | 输出建议 + 等待主人确认 |
| < 50 | 暂缓，仅知识库 |

**审批输出格式**：
```
=== 技能审批结论 ===
技能名: <名称>
评分: <六维得分> = <总分>
路由: <关系分类>
建议动作: <create/patch/跳过>
审批: <自动批准 / 等待主人确认>
```

## 执行步骤

### Step 1: 获取判定输入

从门禁 5 产出中提取：
- 3 问中的「最关键决策」
- 3 问中的「新发现」
- 3 问中的「下轮需记得」
- 本次任务类型标签（编码/配置/调试/规划/知识整理/系统维护/...）
- 计划的目标层级（L1-L5）

### Step 2: 六维评分

对六个维度逐一评分，每个维度附一行理由。

**输出格式**：
```
## 技能化判定 v3
**任务**: {描述}
**目标层级**: L{1-5}

| 维度 | 评分 | 理由 |
|------|:----:|------|
| 问题严重性 (S) | X/10 | ... |
| 独立性 (I) | X/10 | ... |
| 触发词区分度 (D) | X/10 | ... |
| 创建成本 (C) | X/10 | ... |
| 知识成熟度 (M) | X/10 | ... |
| 目标层级 (L) | X/10 | ... |

**加权总分**: {score}/100
**层阈值**: {目标层阈值}
```

### Step 2.5: 路由检查

执行路由节点决策树。

```
### 路由检查
[命中] {无 / 技能名列表}
[关系] {无匹配 / 补充 / 校正 / 修改 / 覆盖 / 不确定}
[建议动作] {create / patch / 跳过 / 等待确认}
```

### Step 3 & 4: 审批 + 最终结论

```
### 审批结论
=== 技能审批结论 ===
技能名: <名称>
评分: <六维得分> = <总分>
路由: <关系分类>
建议动作: <create/patch/跳过>
审批: <自动批准 / 等待主人确认>

### 最终结论
[触发词建议] {3-5 个词}
[目标位置] skills/{layer_label}/{name}/
```

### Step 5: 埋点（可选）

```powershell
# 记录判定结果
memory_write(key="域三：教训库.skill-judge-history.{date}", value="score={score}, route={route}, action={action}")
```

## 何时用 / 何时不用

| 场景 | 用 / 不用 |
|------|-----------|
| 门禁 5 产出发现新模式需要技能化 | ✅ 用 |
| 需要评估已有技能是否需要更新 | ✅ 用 |
| 需要创建新技能 | ✅ 用 |
| 纯文本润色/简单问答 | ❌ 不用 |
| 系统配置修改 | ❌ 不用（gateway/cron 被 deny） |
| 与 audit-skill/反思 相关的审计 | ❌ 不用（走 audit-skill） |

## 反模式

- ? 评分后不记录 — 判定结果应写入 memory 供后续参考
- ? 只给数字不给理由 — 主人需要看懂为什么
- ? 跳过某个维度 — 六个维度必须全部评分
- ? 自行调用 skill_manage — 本技能只判定不创建
- ? 对纯闲聊/只读查询做判定 — 豁免场景直接跳过
- ? 路由判定「覆盖」仍创建 — 路由优先于分数
- ? 用估算代替 D 维度的实际搜索 — 必须实测
