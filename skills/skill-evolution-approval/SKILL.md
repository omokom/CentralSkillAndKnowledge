# 🧬 Skill Evolution & Approval — 技能进化审批机制

## 核心原则

### 三条红线（违反即刻拒绝）
1. **🔴 禁止修改工作流层技能** — 红线。`infraloop-workflow`、`audit-skill` 等 L1 工作流技能不可修改，不可删除。违者直接驳回并上报。
2. **🔴 禁止删除他人技能** — 红线。只能创建新技能或补充自己职责范围内的技能。
3. **🔴 禁止创建破坏性技能** — 红线。含 `gateway`、`cron` 操控或系统配置修改的技能一律拒绝。

### 技能分层

| 层级 | 名称 | 范围 | 修改权限 |
|------|------|------|----------|
| L1 🔒 | 工作流层 | 系统级工作流、审计、沉淀管道 | **完全锁定** |
| L2 📋 | 管理层 | task-planner、quality-gate | 仅组长级 Agent 可申请修改 |
| L3 ✏️ | 专业层 | 各角色的工作技能 | 对应 Agent 可申请创建/修改 |
| L4 🆕 | 实验层 | 新创建的未评级技能 | 任何 Agent 可申请 |

---

## 技能进化工作流

### Phase 1: 发现问题 → 触发评估

当以下情况发生时，触发技能评估：
- 同一个问题反复出现 ≥3 次
- 任务耗时超过预期的 2 倍
- 多个 Agent 使用不同的方法做同一件事
- 发现新的工作模式或规律
- 反思中发现系统性的改进机会

### Phase 2: 结构化评估（六维评分）

```
【技能评估报告】
评估人: <你的角色>
评估日期: <YYYY-MM-DD>

1. 问题严重性（1-10）: _
   - 这个问题出现频率多高？阻塞/效率低/纯信息？
   
2. 独立性问题（1-10）: _
   - 这个问题能否独立解决，还是依赖其他系统修改？
   
3. 触发词区分度（1-10）: _
   - 用 2-5 个词能否精准触发这个技能？
   
4. 创建成本（1-10）: _
   - 写这个 SKILL.md 需要多少工作量？（10=最少）
   
5. 知识成熟度（1-10）: _
   - 我们对该问题的理解是否足够成熟？
   
6. 目标层级（1-10）: _
   - 这个技能属于哪个层级？
     L2(管理层)=6-10, L3(专业层)=3-5, L4(实验层)=1-2

综合评分: (S1+S2+S3+S4+S5+S6) / 6
```

### Phase 3: 路由决策

| 评分区间 | 决策 | 说明 |
|----------|------|------|
| ≥ 8.0 | ✅ 批准创建 | 立即创建技能，自动分配 |
| 6.0 - 7.9 | 📋 提交审批 | 生成审批单，由管理层 Agent 审批 |
| 4.0 - 5.9 | 📝 记录待议 | 记录到技能改进池，周会讨论 |
| < 4.0 | ⏸️ 暂缓 | 条件不成熟，标记待观察 |

**审批流：**
```
创建/修改申请
  → L4 实验层: 自批准（评分≥8）
  → L3 专业层: 同组组长审批
  → L2 管理层: 技术总监/质量总监审批
  → L1 工作流层: ❌ 禁止
```

### Phase 4: 技能创建

审批通过后执行技能创建：

```markdown
---
name: <英文名>
version: 1.0.0
description: "<一句话描述>"
triggers:
  - "<触发词1>"
  - "<触发词2>"
tools:
  - read
  - write
  - edit
  - exec
metadata:
  codex:
    emoji: "<emoji>"
---

# <Emoji> <标题>

## 职责
<明确的职责说明>

## 工作流程
<标准操作步骤>

## 输入与输出
<输入物和输出物>

## 质量标准
<验收条件>
```

**文件路径:** `~/.openclaw/skills/<技能名>/SKILL.md`

### Phase 5: 注册到 Multica

通过 Multica 的 skill 管理功能将新技能导入工作区，然后分配给对应 Agent：

```
exec("multica skill create --workspace-id <wsId> --file <path> 2>&1")
exec("multica agent skills add <agentId> --workspace-id <wsId> --skill-ids <skillId> 2>&1")
```

### Phase 6: 后续追踪

- 新技能创建 7 天后：检查触发频率和使用效果
- 使用频率 = 0 的 30 天后：标记为废弃候选
- 使用频率 > 10 次/周：考虑升级为内置技能

## 审批单模板（评分 6.0-7.9 时使用）

```
=== 技能审批单 ===
申请时间: <ISO 8601>
申请人: <角色名>
技能名: <提议名称>

问题描述:
<明确描述现有技能覆盖不了什么场景>

六维评分: <S1>/<S2>/<S3>/<S4>/<S5>/<S6> = <综合分>

草稿内容:
<初步的 SKILL.md 内容，或关键段落>

审批结果: [待审批/批准/拒绝]
审批人: ___________  日期: ___________
审批意见: <reason>
```

## 失败模式

| 问题 | 处理方式 |
|------|----------|
| 新技能与现有技能重叠 | 合并到现有技能，不新建 |
| 技能评分 < 6 但情况紧急 | 先手工执行方案，补充评分依据后重新评估 |
| 审批超时 24h | 自动升级到上一层审批 |
| 审批拒绝 | 记录拒绝理由，7 天内不得就同一问题再次提交 |
