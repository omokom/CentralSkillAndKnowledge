# 💾 Knowledge Precipitator — 知识沉淀

## 职责

任务完成后，将有价值的经验、教训、模式提炼为结构化知识条目，写入知识库。确保知识可复用、可检索。

## 沉淀对象

| 类型 | 条件 | 目标路径 |
|------|------|----------|
| 经验技巧 | 发现新的有效工作方法 | `knowledge/workspace/经验/<分类>/` |
| 教训 | 错误/踩坑/返工 | `knowledge/workspace/教训/<分类>/` |
| 模式 | 发现可复用的流程/模板 | `knowledge/workspace/模式/<分类>/` |
| 参考 | 有价值的外部资料/配置 | `knowledge/workspace/参考/<分类>/` |
| 反思 | 任务执行回顾 | `knowledge/workspace/反思/` |

## 沉淀工作流

### 阶段 1：识别沉淀价值
检查任务产出中是否有以下信号：
- 耗时 >30min 才解决的问题 → 应该沉淀
- 尝试 ≥3 种方案才成功的 → 应该沉淀
- 可能被其他角色复用的 → 应该沉淀
- 重复出现 2 次以上的问题模式 → 必须沉淀
- 纯常规任务、无新信息 → 跳过

### 阶段 2：提取结构化内容
按以下模板组织沉淀内容：

```markdown
# <分类> - <标题>

## 场景
在什么任务/情况下发现的？

## 结论
核心要点是什么？（1-3句话）

## 详细内容
具体步骤、代码、配置、注意事项

## 来源
- 任务编号: <TASK-xxx>
- 涉及角色: <你自己>
- 日期: <YYYY-MM-DD>
```

### 阶段 3：写入知识库
```
write("knowledge/workspace/<分类>/<子分类>/<YYYY-MM-DD>--<标题>.md", "<沉淀内容>")
```

### 阶段 4：更新索引
将新条目追加到知识库索引文件：
```
exec("echo \"| <日期> | <分类> | <标题> | <角色> |\" >> knowledge/workspace/README.md")
```

## 沉淀质量标准

- [ ] 内容可独立理解（不看原始任务也能看懂）
- [ ] 包含具体参数/配置/代码片段（如有）
- [ ] 标注了适用场景和边界条件
- [ ] 文件名和路径符合命名规范
- [ ] 索引已更新
- [ ] 每条沉淀 ≤ 500 字（太长没人看）

## 禁止行为

- 不沉淀普通配置值和已知常识
- 不沉淀敏感信息（密码、token、私钥）
- 不沉淀个人临时偏好（沉淀有价值的经验而非随意决定）
- 不重复沉淀已有知识（先搜索确认）
